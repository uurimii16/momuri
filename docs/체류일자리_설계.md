# 체류·일자리 기능 설계 — momuri Phase 2 (과제4 핵심)

> 2026 관광데이터 공모전 ②-2 과제4의 정체성("일회성 관광 → 체류·생활인구 유입")을 실체화하는 핵심 기능 설계.
> 상위 기획: [`웹앱_기획_과제4.md`](웹앱_기획_과제4.md) · (작성: 2026-07-07)

## 0. 결정된 방향
- **MVP 주인공: 2030 워케이션·한달살기(장기 체류)** — 공고 과제4 공식 예시("체류형 로컬 관광 플래너", #워케이션 #한달살기 #2030청년 #로컬관광)와 정렬.
- 장기 체류이므로 **로컬 일자리·장기 숙소가 곁가지가 아닌 핵심**. "문화유산 매력으로 유입 → 체류형 코스 → 로컬 일자리·워케이션으로 생활인구" 한 흐름.

## 1. 사용자 흐름
```
[체류 조건 입력]
  · 체류 기간: 1박2일 / 1주 / 한 달+   (기본=한 달)
  · 관심 테마: 문화유산(MVP)
  · 선호 직무(선택): 예) 카페·농업·사무·원격근무
        ↓
[체류형 코스]   Day1 하루코스 → 야간=장기 숙소 → Day2 … (기간만큼)
[로컬 일자리]   해당 시군구 채용 카드(회사·직무·고용형태·급여·마감·링크)
[장기 숙소]     주/월 단위 숙박(게스트하우스·펜션·한달살기 숙소)
[생활인구 스토리] 방문자수·인구감소지역 데이터로 "머물면 생기는 효과"
```

## 2. 데이터 레이어 (analysis/ → map/data/*.json, 정적 배포)
> Netlify 정적 사이트라 서버 없음 → **키 노출 방지 위해 오프라인 배치로 JSON을 구워서 커밋**(기존 momuri.json/courses.json과 동일 패턴). 실시간성은 주기적 재생성으로.

### 2.1 장기 숙소 — 기존 TourAPI 재사용 (신규 API 불필요, 키 보유) ✅구현
- `build_service_data.py`에 `contentTypeId=32`(숙박) 수집 추가 → `momuri.json` 각 지역에 `stays[]`. (2026-07-07 구현·검증)
- 필드: title·addr·lat·lng·img·id. (상세 등급/장기여부는 `detailIntro2` 보강 여지)
- ⚠️ **소도시 숙박 희소**(라이브 확인): 문경 4건 있으나 **상주 0건**. 축제와 동일 패턴. → 체류 코스(step 4)에서 시군구 0건이면 **지역 중심좌표 반경검색(`locationBasedList2`, 예 20km)** 으로 인근 숙소 폴백. 반경 폴백은 도 단위 폴백보다 숙박에 적합.

### 2.2 로컬 일자리 — 지표 + 딥링크 + 잡알리오 (개인 계정 가능 조합)
> ⚠️ **워크넷 채용정보 API(3038225)는 폐기**: 실제 구인 목록/상세는 **기업·기관 회원 전용**이라 개인 계정 불가(2026-07-07 확인). 공고상 외부 일자리 API는 **선택**(필수는 공사 TourAPI뿐)이므로, 개인이 쓸 수 있는 아래 조합으로 대체 — 오히려 moat(데이터 근거)와 더 정합.
> (직업정보·직무데이터사전(15088880)·직업전망(3071367)은 "직업 설명"용 → 우리 것 아님. 혼동 주의.)

**(A) 일자리 지표 — 데이터 근거(moat)**: 지역별 **구인배수·인력부족률·청년 순유출** 등으로 "이 소도시는 일손이 부족하다"는 생활인구 논거. 출처: KOSIS/고용노동통계(개인 가능). 지역별 `job_index.json`. → #생활인구
**(B) 실물 채용 딥링크 — 항상 최신·키 불필요**: "이 지역 채용 보기" 버튼 → 워크넷/고용24 **지역 채용검색 결과 URL 딥링크**(지역코드 파라미터). API 게이트 우회. → #로컬일자리
**(C) 잡알리오 공공기관 채용 — 개인 자동승인 API**: `기획재정부_공공기관 채용정보 조회서비스`(data.go.kr **15125273**, 또는 opendata.alio.go.kr). 지역 소재 공공기관 채용 카드(제목·직무·인원·접수기간·링크). 신규 `build_jobs.py` → `jobs.json`. 키=env `JOBALIO_KEY`. → #로컬일자리 보조
> 흐름: "일손 부족(A 지표) → 워케이션하며 채용 보기(B 딥링크 + C 공공기관 카드) → 머물면 생활인구".

### 2.3 체류형(다일) 코스 — 기존 코스 확장
- `build_courses.py`의 하루 5스톱 그리디 동선을 재사용해 **N일 반복 구성**.
- 야간 스톱 = `stays[]` 최근접 숙소. 코스 스키마에 `day` 필드 추가(courses.json 하위호환 유지: 기존 1일치는 day=1).

## 3. 프론트 (map/)
- 신규 **`stay.html`**(또는 course.html 확장): 상단 체류조건 토글 → ①체류형 코스 타임라인(day 탭) ②로컬 일자리 패널 ③장기 숙소 카드 ④생활인구 스토리 박스.
- 진입: 랜딩 카드 + momuri.html 지역 팝업에 "머무르기" 버튼.
- 기존 밝은 테마·로고·TourAPI 출처 표기 재사용.

## 5. 구현 순서 (권장)
1. **[완료] 장기 숙소**: `build_service_data.py`에 contentTypeId=32 추가. → momuri.json 재생성 필요.
2. **[지금·키불필요] 채용 딥링크(B)**: 워크넷/고용24 지역 채용검색 URL 패턴 → 프론트 "이 지역 채용 보기" 버튼.
3. **[사용자 액션] 잡알리오 신청**(data.go.kr 15125273) → `JOBALIO_KEY` → `build_jobs.py` → jobs.json(C).
4. **일자리 지표(A)**: 구인배수·청년고용 지역 데이터(KOSIS/고용통계) → `job_index.json`. (소스 확정 후, 필요 시 KOSIS 키 안내)
5. `build_courses.py` 다일 확장(day 필드) + 숙박 야간 스톱(0건 시 반경 폴백) → courses.json.
6. `stay.html` 프론트(체류조건 → 다일 코스 + 일자리 지표/딥링크/잡알리오 + 숙소 + 생활인구 스토리) + 진입 동선.

## 6. 미해결·리스크
- 워크넷 지역코드 매핑 정확도(시군구명 표기 차이) — 공통코드 응답으로 대조 검증.
- 쇠퇴 소도시는 채용 공고도 적을 수 있음 → 0건 시 인근/원격근무·도 단위 폴백(축제와 동일 패턴).
- 장기 숙소 "한달살기" 명시 데이터는 부족 → 일반 숙박 + "장기 문의" 안내로 시작.

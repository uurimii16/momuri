# 웹·앱 부문 기획 — 과제4 (쇠퇴지 중심 관계인구 서비스)

> 2026 관광데이터 활용 공모전 ②-2 웹·앱 구현 부문 지정과제 **4번** 출품 기획.
> 해결과제: *"단순 흥미 위주 일회성 관광의 한계 및 인구 감소 지역의 지속가능한 생활인구 유입 대책 부족"*
> 포스터 부문(`tourgo-tourism-decline`)의 분석 엔진을 서비스로 재활용 → **한 데이터로 두 부문**.
> (작성 시작: 2026-07-06)

---

## 0. 한 줄 정의

> **데이터가 지목한 우선 쇠퇴지에 "지금 갈 이유"를 심고, 반복 방문·체류로 이어 관계인구·생활인구로 전환하는 서비스.**

일반 여행앱은 인기지역을 추천한다. 이 서비스는 **복합쇠퇴지수가 진단한 지역만** 대상으로, "왜 굳이 거길?"이라는 동기 문제를 **이벤트/훅**으로 푼다.

## 1. 왜 이 프레임인가 (전략)

- **과제2(오버투어리즘 분산)의 한계**: 주인공=관광객, 쇠퇴지=조연('붐비는 곳 대신 가는 조용한 대안'). 우리 moat(쇠퇴진단)가 조연이 됨.
- **과제4 전환**: 주인공=쇠퇴지. moat·포스터 사명(관계인구·지방소멸 대응)과 정렬.
- **동기 문제**: 아무도 "쇠퇴한 군에 가야지" 하고 일어나지 않음 → **당길 힘(pull) = 이벤트/훅**이 필수. 이게 서비스의 심장.

## 2. 핵심 서사 — 3단 퍼널

과제4 문구("일회성의 한계 → 지속가능한 생활인구 유입")를 그대로 구현:

| 단계 | 훅 | 사용자에게 주는 것 | 데이터 |
|---|---|---|---|
| ① **갈 이유** (일회성) | 이벤트·한정체험 캘린더 (시간성) | "그 지역에서·그 시기에만" | TourAPI 축제/행사, 국가유산 행사(야행·생생) |
| ② **다시 올 이유** (관계인구) | 스탬프·디지털 관광주민증형 리워드 | 반복 방문·수집·환원 | 방문 인증, 지역화폐/관광주민증 연계 |
| ③ **머무를 이유** (생활인구) | 체류·워케이션 코스 | 장기 체류 동선 | 장기숙소, 워케이션 거점, 로컬 일자리 |

- 근거: 디지털 관광주민증 시범(강원 평창·충북 옥천, 2022) 10개월 8만 명 발급 → 관광소비 증대 [권태일·장유진, 2023, 투어고 2023-6호]. **이미 검증된 관계인구 전환 메커니즘.**

## 3. moat — 남들과 다른 이유

프롬프톤 경쟁작 다수는 "LLM한테 추천받기". 우리는 **실데이터 엔진**이 3가지를 결정:

1. **어디를**: 복합쇠퇴지수 3축(방문·매력·연결)이 우선 쇠퇴지 지목 → 아무 데나 아님
2. **어떻게 잇나**: 동선 네트워크 중심성 → 소외지를 연계 루트로
3. **뭘로 당기나**: 국가유산 15,728건 좌표 + 미식 논거 → 그 지역만의 콘텐츠

## 4. 타깃 사용자

- 1차: **관계인구 지향 여행자** (반복 방문·로컬 경험 선호). 과제4 예시의 2030 워케이션·한달살기층 포함하되 더 넓게.
- 확장: 인바운드(외국인) — 다국어 기반(map 프론트에 KO/EN/ZH/JA 완비) → 과제3(K-콘텐츠)와 연결.
- B2G 시선: 인구감소지역 89 지자체가 "관계인구 유입 성과"를 보는 대시보드로 확장(→ 과제10과 연결).

## 5. 데이터/API 레이어

### ⚠️ 대전제: 한국관광공사 OpenAPI **필수** (공고 규정)
- *"최종 완성 서비스 내 한국관광공사 OpenAPI를 필수로 활용"* — 미충족 시 **심사 제외**.
- 심사 배점: "데이터 활용 적절성(공사 OpenAPI 활용 필수) **20점**" 별도.
- → **서비스 뼈대의 각 단계를 공사 TourAPI로 직결**한다. 그 외 API는 "외부 추가 활용"으로 허용됨.

### 5.1 공사 OpenAPI = 서비스 축 (필수)

| 단계 | 공사 OpenAPI | 용도 |
|---|---|---|
| **어디를**(쇠퇴 진단) | 한국관광 **데이터랩 지역별 방문자수**(B551011, data.go.kr 15101972) | 방문 추세·혼잡 → 쇠퇴지수 방문축(이미 파이프라인 보유) |
| **갈 이유**(이벤트) | TourAPI **행사정보 `searchFestival2`** | 시군구·기간별 축제/공연/행사 = "지금 갈 이유" |
| **콘텐츠 스톱**(볼·먹·체류) | TourAPI **`areaBasedList2` / `locationBasedList2` / `detailIntro2`** | 시군구코드별 관광지·음식·숙박·체험 |
| **다국어**(인바운드) | TourAPI **다국어 서비스(Eng/Chs/Jpn)** | 외국인 대상(지도 KO/EN/ZH/JA 완비) |
| **키워드 검색** | TourAPI **`searchKeyword2`** | 관심사·POI 검색 |

> 접수·API 발급: 한국관광콘텐츠랩(api.visitkorea.or.kr) 회원가입 → OpenAPI 활용신청.

### 5.2 우리 고유 엔진 (moat, 오프라인 분석 → API로 노출)
- 복합쇠퇴지수 3축·활성화 타깃 16, 동선 네트워크 중심성, 국민여행조사 재방문·추천의향.
- 공사 API가 "무엇이 있나"를 준다면, 우리 엔진은 **"어디가 우선 쇠퇴지이고 어떻게 잇나"**를 결정 → 차별점.

### 5.3 외부 API 추가 (선택, 공고상 자유 허용)
| 레이어 | 추가 |
|---|---|
| 문화유산 | 국가유산청 검색 API(좌표·나레이션 4개국어, 밀도 15,728건) — 콘텐츠 보강 |
| 타깃 정합 | 행안부 **생활인구·인구감소지역 89** 통계(효과측정) |
| 경로 | Routo(지도), 대중교통(ODsay/TAGO) |
| 리워드 | 방문 인증(위치), 지역화폐/관광주민증 연계 |
| 내러티브 | LLM(관심사 큐레이션·다국어·상황 맞춤) |

## 6. 핵심 루프 (입력 → 출력)

```
[입력] 관심사(미식/문화유산/체류) + (선택)기간·동행
   │
① 타깃 선정  ── 쇠퇴지수·동선으로 관심사에 맞는 우선 쇠퇴지 큐레이션
② 갈 이유    ── 그 지역의 지금/이번 시즌 이벤트·한정체험 (시간성)
③ 코스 구성  ── 문화유산·미식 스톱 + Routo 동선, LLM 다국어 내러티브
④ 관계 훅    ── 방문 스탬프 적립 → 반복방문 리워드 → 체류(워케이션) 제안
   │
[출력] '이 쇠퇴지에 지금 갈 이유' + 스탬프북 + 체류 코스
```

## 7. 화면 (기존 `map/app.js` 재활용)

- 기존 지도 셸·다국어·상세패널 재사용, 탭에 **"관계인구 코스"** 추가
- 쇠퇴 타깃 지도 마커 → 클릭 → [지금 이벤트 / 체험 / 체류옵션 / 스탬프]
- **마이 스탬프북** — 반복 방문 게이미피케이션(관계인구 시각화)
- 상세패널: 쇠퇴지수·문화유산·미식·이벤트 일정

## 8. 확장 로드맵

- **Phase 1 (보유 자산 MVP)**: 쇠퇴 타깃 + 국가유산 콘텐츠 + 지도 + 스탬프(로컬 목업). 데이터 근거 완결 → 심사 데모 성립.
- **Phase 2 (시간성)**: TourAPI 축제/행사 + 국가유산 행사 → "지금 갈 이유" 실동작.
- **Phase 3 (관계→생활)**: 생활인구 통계 결합, 체류/워케이션 코스, 리워드 정교화, 인바운드 다국어.

## 9. 포스터 부문 시너지

`analysis/` 엔진(쇠퇴지수·네트워크·문화유산)을 API로 노출 → 웹앱이 호출.
**포스터=분석근거 / 웹앱=서비스**, 한 데이터로 두 부문. 심사 스토리 강화.

## 9-A. 심사기준 매핑 (공고 기준, 어디서 점수 따나)

**1차 심사(100)**
- 서비스 구현성 30 — **작동하는 완성 서비스 필수**(목업 불가). 과제4 연계기능(진단→이벤트→코스→리워드) 완결·안정.
- 서비스 기획력 30 — 쇠퇴지수 moat + 3단 퍼널 서사(독창성).
- 데이터 활용 적절성 **20** — 공사 TourAPI를 뼈대에(5.1). *여기서 감점 안 나게 하는 게 최우선.*
- 서비스 발전성 20 — 8장 확장 로드맵(Phase 1→3), 인바운드·B2G 확장.

**최종 PT(100)**: 적정성 30 / 완성도 30 / 실용성 25 / 발표 15 → 실제 구현도·데이터활용·안정성 비중 큼.

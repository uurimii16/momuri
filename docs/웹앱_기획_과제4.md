## 0. 한 줄 정의

> **문화유산 매력으로 "숨은 로컬 매력지(쇠퇴 소도시)"를 발견하게 하고, 하루→체류형 코스로 현실화하며, 공사 관광데이터로 채우고, 로컬 일자리·워케이션까지 연결해 일회성 관광을 체류·재방문(생활인구)으로 전환하는 체류형 로컬 관광 플래너.**

**과제4 정체성 = "일회성 관광 → 체류·생활인구 유입".** 문화유산 매력은 *유입 훅*(왜 가는가), **체류·재방문·로컬 일자리 매칭이 MVP 핵심**(어떻게 머무는가). 관광 활성화와 생활인구 유입은 대립이 아니라 한 흐름의 앞·뒤.

## 1. 전략 — 과제4를 정면으로

- 이 부문 결과물 = **작동하는 웹/앱 서비스**(분석·보고서·정책제안 아님). 심사도 "완성 서비스"만 대상.
- 과제4 해결과제 = *"단순 흥미 위주 일회성 관광의 한계 + 인구감소지역 생활인구 유입 대책 부족"*. → 서비스가 **일회성 방문을 체류·재방문으로 바꾸고, 그것이 생활인구 유입으로 이어지는 경로**를 실제 기능으로 보여야 함.
- 공사 대회지만 과제4 예시(체류형·워케이션·로컬 일자리)가 방향을 못박음 → **관광 매력(유입) + 체류·일자리(정착)를 둘 다** 구현해야 정합. 문화유산은 "왜 이 소도시에 가나"의 답, 체류·일자리는 "왜 머무나"의 답.
- 차별점: 경쟁작이 LLM 추천 플래너로 "발견"에 그칠 때, 우리는 **데이터 엔진(쇠퇴지수·동선)으로 유입지를 고르고 + 로컬 일자리·체류로 관계인구까지** 잇는다.

## 2. 서비스 개념

### 2.1 매칭 로직 (매력 먼저, 동선은 거들기)
```
사용자 관심(테마: ★문화유산[유입 훅] / 한류·자연·미식[Phase 2])
  → 그 테마를 진짜 충족하는 '숨은 매력 쇠퇴지' 매칭        [유입] 왜 가나
  → 동선 네트워크로 하루 코스 현실화(경유·이동)            [경험] 하루
  → 공사 TourAPI로 콘텐츠(명소·먹거리·축제·숙박) 채움       필수 데이터
  → 체류 조건(기간·선호 직무) 입력 → 체류형(1박+·한달살기) 코스   ★MVP 핵심
  → 로컬 일자리·워케이션 + 장기 숙소 동선 결합 → 생활인구        ★MVP 핵심 [정착] 왜 머무나
```
> ⚠️ 유입만으론 과제2(숨은명소 발굴)에 그침. **머무는 이유(체류·일자리)를 실제 기능으로** 넣어야 과제4. 매력(문화유산)은 진입점, 체류·일자리가 종착.

### 2.2 3층 구조
| 층 | 내용 | 노출 |
|---|---|---|
| **① 토대(moat)** | 쇠퇴지수로 활성화 우선 소도시 선정 + 테마 유형 태깅 + 동선 네트워크(어느 허브에서 어떻게 연결/소외되나) | 오프라인 분석 → 데이터로 앱에 내장(사용자는 근거로만) |
| **② 서비스(★심사 main★)** | 관심 입력 → 숨은 매력지 코스 제시 → 공사 API 콘텐츠 → 체류·재방문·일자리 매칭 | 사용자가 쓰는 앱 |
| **③ 임팩트(스토리)** | 소외지 방문↑=관광 활성화 / 반복·체류=생활인구 / 일자리 연결=지방기업 인력난 → 청년↔지역 3면 | 기획력·발전성 점수 |

## 3. "숨은 로컬 매력지" 정의

**쇠퇴 + 테마자산 있음 + 소외됨(안 알려짐).**
- 자산 없는 극단 쇠퇴지(예: 울릉 — 멀고 자산 적음)는 **안 밀어냄**(정직한 필터).
- 자산은 있는데 동선에서 소외된 곳(문화유산형 타깃 등)만 → 사용자에겐 "진짜 가고 싶은 곳", 지역엔 "부당하게 소외된 곳".
- 포스터의 **타깃 유형화(문화유산형/자연힐링형)**와 직결.

## 4. moat — 남들과 다른 이유
1. **쇠퇴지수**: 어디가 활성화 우선지인가 (남들: 없음)
2. **동선 네트워크**: 매력지를 현실적 코스로 + 소외 진단 (남들: 데이터 없음)
3. **테마 유형화 + 미식·문화유산·한류 논거**: 지역별 관광 특화 (포스터 자산)
→ 경쟁작 다수는 "LLM 추천 플래너". 우리는 데이터 엔진이 추천의 근거.

## 5. 데이터 / API

### ⚠️ 대전제: 한국관광공사 OpenAPI **필수**(미충족 시 심사 제외, 배점 20점)
- ✅ **키 발급·활성화 완료**(2026-07-06): 국문 관광정보 TourAPI 4.0 `KorService2`. 행사정보(`searchFestival2`)는 이 서비스에 **포함**(별도 신청 불필요).
- ⚠️ **축제 데이터 특성(2026-07-07 라이브 검증)**: `searchFestival2`는 (1)쇠퇴 소도시는 등록 축제가 거의 없고(상주=0건), (2)차년도(2026) 축제가 API에 늦게 반영됨 → "오늘 이후"만 조회하면 도(道) 단위도 0건. (경북: `eventStartDate=2025-01-01`→23건, `2026-01-01`→0건 확인). **해결(코드 반영·검증 완료)**: 최근 1년 창으로 조회해 연례 축제까지 포착하고 `status`(upcoming/annual)·`scope`(sigungu/sido) 태깅 → 시군구 0건이면 도 단위 폴백. 결과: 상주→경북 인근 연례 축제 10건, 문경→문경사과축제 노출. 2026 일정이 등록되면 자동으로 `upcoming` 승격·상단 정렬. 프론트는 `annual`을 "매년 N월 열리는 연례 축제(올해 일정 확인 필요)", `sido`를 "인근 지역 축제"로 표기 예정. → "지금 갈 이유"는 상설 문화유산·명소가 주, 축제는 보조.

## 6. 화면 — 구현 현황 (배포: momuri.netlify.app)
- 루트 랜딩 `index.html`(4페이지 진입) + 페이지 간 내비(홈 탭·로고 클릭).
- ✅ **`momuri.html`** — 숨은 매력지 12곳 지도. POI 핀 클릭 팝업(사진·카테고리·주소), 마커 색상 범례, 공사 TourAPI 콘텐츠.
- ✅ **`course.html`** — **하루 코스 타임라인**(신규 완성). 명소·맛집을 하루 리듬으로 묶은 동선 + 지도 실도로 경로선 + **자동차/대중교통 토글**(버스번호·환승·구간분, ODsay). 데이터 `courses.json`(12코스).
- ✅ **`decline-map.html`** — 시군구 쇠퇴지수 단계구분도(Jenks). 클릭 시 **방문·매력·연결 3축 분해 + 주된 쇠퇴 요인** 표시(지수 차별점 증명), 쇠퇴 Top15 랭킹(flyTo), GeoJSON 경량화(9.2MB→0.67MB).
- ✅ **`methodology.html`** — 지표 방법론 설명서.
- 밝은 테마 + 머무리 로고(인라인 SVG) + 한국관광공사 TourAPI 출처 표시.

## 9. 개발 로드맵 (마감 7/21=신청·기획서 / 기능 완성 ~9월 / 기능심사 10월)
- **Phase 1** ✅(완료·배포): 숨은 매력지(문화유산형) 매칭 + 하루 코스(자동차/대중교통) + 공사 API 콘텐츠 + 쇠퇴지수 지도.
- **Phase 2 = 과제4 MVP 핵심**(개발기간 7말~9월, 기능심사 대상): ★**체류형 코스**(1박+·한달살기, 체류 기간·선호 조건 입력) ★**로컬 일자리·워케이션 매칭**(워크넷/고용24 채용 API + 장기 숙소 동선 결합) ★**생활인구 연결 스토리**(방문자수·인구감소지역 데이터). + 축제 0건 폴백 해결, 지도 좌표 전국 확장.
- **Phase 3**(발전성): 인바운드 다국어, 미식/한류/자연 테마 확장, 활성화 효과 대시보드(방문자수 API), 재방문·관계인구 트래킹.

> 순서 근거: 과제4 정체성(체류·일자리)이 Phase 2 = 기능심사(10월) 필수. 미식 등 테마 확장은 과제4 핵심이 아니므로 발전성 레이어(Phase 3)로 후순위.

## 10. 과제4 앵커 (과제2와의 차이)
